# 서론

스킬(Skill)은 Claude에게 특정 작업이나 워크플로우를 처리하는 방법을 알려주는 지침을 폴더 하나로 묶어놓은 것입니다. Claude를 내 필요에 딱 맞게 커스터마이징하는 가장 강력한 방법이기도 합니다. 대화할 때마다 선호하는 방식, 프로세스, 도메인 지식을 반복 설명할 필요 없이, 스킬로 한 번만 가르쳐두면 이후부터 바로 활용할 수 있습니다.

스킬은 반복되는 워크플로우에서 진가를 발휘합니다. 명세서를 바탕으로 프론트엔드 디자인을 만들거나, 일관된 방법론으로 리서치를 수행하거나, 팀 스타일 가이드에 맞춰 문서를 작성하거나, 여러 단계의 프로세스를 자동화하는 경우가 대표적입니다. 코드 실행, 문서 생성 같은 Claude의 내장 기능과도 잘 어울립니다. MCP 통합을 구축하고 있다면, 스킬이 한 겹 더해져서 단순한 도구 접근을 신뢰할 수 있는 최적화된 워크플로우로 끌어올려 줍니다.

이 가이드는 효과적인 스킬을 만드는 데 필요한 모든 내용을 다룹니다. 기획과 구조부터 테스트와 배포까지 전부 포함됩니다. 개인용이든, 팀용이든, 커뮤니티 공유용이든 상관없이, 실용적인 패턴과 실제 사례를 곳곳에서 확인할 수 있습니다.

### 이 가이드에서 배울 내용

- 스킬 구조의 기술적 요구사항과 모범 사례
- 독립형 스킬과 MCP 기반 워크플로우를 위한 패턴
- 다양한 사용 사례에서 검증된 패턴
- 스킬을 테스트하고, 반복 개선하고, 배포하는 방법

### 대상 독자

- Claude가 특정 워크플로우를 일관되게 따르도록 하고 싶은 개발자
- Claude로 반복 작업을 자동화하고 싶은 파워 유저
- 조직 전체에서 Claude 사용 방식을 표준화하려는 팀

### 이 가이드를 활용하는 두 가지 경로

독립형 스킬을 만드시나요? **기본 원리, 기획과 설계, 카테고리 1-2**에 집중하세요. MCP 통합을 강화하시나요? **"스킬 + MCP" 섹션과 카테고리 3**을 보시면 됩니다. 두 경로 모두 기술적 요구사항은 동일하니, 본인의 사용 사례에 맞는 부분을 골라 읽으세요.

### 이 가이드를 마치면

가이드를 끝까지 따라가면, 한 자리에서 바로 쓸 수 있는 스킬을 직접 만들 수 있게 됩니다. skill-creator를 활용하면 첫 스킬을 만들고 테스트하는 데 **15~30분**이면 충분합니다.

바로 시작해 봅시다.

# Chapter 1: 기본 원리

## 스킬이란 무엇인가?

![skill-guide-infographic-01-structure](https://raw.githubusercontent.com/joonlab/md-share-db/main/images/skill-guide-infographic-01-structure_d0042e.png)

스킬은 아래 항목으로 구성된 폴더입니다:

- **SKILL.md** (필수): YAML frontmatter가 포함된 Markdown 지침 파일
- **scripts/** (선택): 실행 가능한 코드 (Python, Bash 등)
- **references/** (선택): 필요할 때 로드되는 참고 문서
- **assets/** (선택): 출력에 쓰이는 템플릿, 폰트, 아이콘

## 핵심 설계 원칙

### 점진적 공개 (Progressive Disclosure)

스킬은 세 단계로 정보를 노출합니다:

- **1단계 (YAML frontmatter)**: 항상 Claude 시스템 프롬프트에 로드됩니다. 전체 내용을 컨텍스트에 올리지 않고도, Claude가 각 스킬을 언제 써야 하는지 판단할 수 있을 만큼의 정보를 담고 있습니다.
- **2단계 (SKILL.md 본문)**: Claude가 해당 스킬이 현재 작업에 필요하다고 판단하면 로드됩니다. 전체 지침과 가이드가 여기에 들어 있습니다.
- **3단계 (링크된 파일)**: 스킬 디렉토리에 함께 번들된 추가 파일입니다. Claude가 필요할 때만 탐색해서 사용합니다.

이 점진적 공개 방식 덕분에 전문적인 역량은 유지하면서 토큰 사용량은 최소화할 수 있습니다.

![skill-guide-infographic-02-progressive-disclosure-v2](https://raw.githubusercontent.com/joonlab/md-share-db/main/images/skill-guide-infographic-02-progressive-disclosure-v2_7ce31a.png)

### 조합성 (Composability)

Claude는 여러 스킬을 동시에 로드할 수 있습니다. 따라서 스킬을 만들 때는 다른 스킬과 함께 쓰일 수 있다는 점을 고려해야 하며, 자기 혼자만 사용된다고 가정하면 안 됩니다.

### 이식성 (Portability)

스킬은 Claude.ai, Claude Code, API 어디서든 동일하게 동작합니다. 한 번 만들어두면 별도 수정 없이 모든 환경에서 쓸 수 있습니다. 단, 스킬이 필요로 하는 종속성을 해당 환경에서 지원해야 합니다.

## MCP 빌더를 위한 안내: 스킬 + 커넥터

> MCP 없이 독립형 스킬만 만드시나요? "기획과 설계"로 바로 넘어가세요. 나중에 언제든 돌아올 수 있습니다.

이미 작동하는 MCP 서버가 있다면, 어려운 부분은 끝난 셈입니다. 스킬은 그 위에 얹는 지식 레이어로, 여러분이 이미 알고 있는 워크플로우와 모범 사례를 담아 Claude가 일관되게 적용하도록 만들어줍니다.

### 주방에 비유하면

- **MCP는 전문 주방**: 도구, 재료, 장비를 제공합니다
- **스킬은 레시피**: 좋은 결과물을 만들기 위한 단계별 지침입니다

이 둘이 합쳐지면, 사용자가 모든 단계를 직접 알아낼 필요 없이 복잡한 작업을 처리할 수 있습니다.

### MCP와 스킬이 함께 작동하는 방식

| MCP (연결성) | 스킬 (지식) |
|---|---|
| Claude를 외부 서비스(Notion, Asana, Linear 등)에 연결 | Claude에게 서비스를 효과적으로 쓰는 방법을 알려줌 |
| 실시간 데이터 접근과 도구 호출을 제공 | 워크플로우와 모범 사례를 담아둠 |
| Claude가 **무엇을** 할 수 있는지 정의 | Claude가 **어떻게** 해야 하는지 정의 |

![skill-guide-infographic-03-mcp-vs-skills](https://raw.githubusercontent.com/joonlab/md-share-db/main/images/skill-guide-infographic-03-mcp-vs-skills_801d9f.png)

### 스킬이 MCP 사용자에게 중요한 이유

**스킬 없이:**

- 사용자가 MCP를 연결해도 다음에 뭘 해야 할지 모릅니다
- "이 통합으로 X는 어떻게 하나요?" 같은 지원 티켓이 쏟아집니다
- 매번 대화가 처음부터 다시 시작됩니다
- 사용자마다 프롬프트 방식이 달라서 결과가 들쭉날쭉합니다
- 진짜 문제는 워크플로우 가이드 부재인데, 사용자는 커넥터 탓을 합니다

**스킬과 함께:**

- 미리 만들어둔 워크플로우가 필요할 때 자동으로 활성화됩니다
- 도구 사용이 일관되고 안정적입니다
- 모든 상호작용에 모범 사례가 녹아 있습니다
- 통합 도구의 학습 곡선이 낮아집니다

# Chapter 2: 기획과 설계

## 사용 사례부터 시작하라

코드를 작성하기 전에, 스킬로 해결할 **구체적인 사용 사례 2-3개**를 먼저 정의하세요.

### 좋은 사용 사례 정의 예시

**사용 사례: "프로젝트 스프린트 계획"**

- **트리거**: 사용자가 "이번 스프린트 계획 좀 도와줘" 또는 "스프린트 태스크 만들어줘"라고 말할 때
- **단계**:
    1. Linear에서 현재 프로젝트 상태를 가져온다 (MCP를 통해)
    2. 팀의 속도(velocity)와 수용 능력(capacity)을 분석한다
    3. 태스크 우선순위를 제안한다
    4. 적절한 라벨과 추정치가 포함된 태스크를 Linear에 생성한다
- **결과**: 태스크가 생성된 완전히 계획된 스프린트

### 스스로에게 물어보세요

- 사용자가 궁극적으로 원하는 결과는 무엇인가?
- 그 결과를 얻기 위해 어떤 다단계 워크플로우가 필요한가?
- 어떤 도구가 필요한가 (내장 도구인가 MCP인가)?
- 어떤 도메인 지식이나 모범 사례를 내장해야 하는가?

## 일반적인 스킬 사용 사례 카테고리

![skill-guide-infographic-04-categories](https://raw.githubusercontent.com/joonlab/md-share-db/main/images/skill-guide-infographic-04-categories_8efd7b.png)

Anthropic이 파악한 대표적인 스킬 사용 사례는 세 가지입니다.

### 카테고리 1: 문서 및 에셋 생성

문서, 프레젠테이션, 앱, 디자인, 코드 등 일관되고 품질 높은 결과물을 생성합니다.

**실제 사례**: `frontend-design` 스킬, `docx`, `pptx`, `xlsx`, `ppt` 관련 스킬 — 높은 디자인 품질로 차별화된 프로덕션 수준의 프론트엔드 인터페이스를 생성합니다.

**핵심 기법:**

- 내장된 스타일 가이드 및 브랜드 표준
- 일관된 출력을 위한 템플릿 구조
- 최종 확정 전 품질 체크리스트
- 외부 도구가 필요 없음 — Claude의 내장 기능을 사용

### 카테고리 2: 워크플로우 자동화

일관된 방법론이 중요한 다단계 프로세스를 자동화하며, 여러 MCP 서버를 조율합니다.

**실제 사례**: `skill-creator` 스킬 — 새 스킬을 만드는 인터랙티브 가이드입니다. 사용 사례 정의부터 frontmatter 생성, 지침 작성, 검증까지 단계별로 안내합니다.

**핵심 기법:**

- 검증 게이트가 있는 단계별 워크플로우
- 일반적인 구조를 위한 템플릿
- 내장된 리뷰 및 개선 제안
- 반복적 개선 루프

### 카테고리 3: MCP 향상

MCP 서버의 도구 접근에 워크플로우 가이드를 더해 활용도를 높입니다.

**실제 사례**: `sentry-code-review` 스킬 (Sentry 제공) — Sentry MCP 서버의 에러 모니터링 데이터를 활용해 GitHub Pull Request에서 감지된 버그를 자동 분석하고 수정합니다.

**핵심 기법:**

- 여러 MCP 호출을 순차적으로 조율
- 도메인 전문성 내장
- 사용자가 직접 입력해야 했을 컨텍스트를 자동 제공
- 일반적인 MCP 이슈에 대한 에러 처리

## 성공 기준 정의

### 스킬이 잘 작동하는지 어떻게 알 수 있을까?

아래 기준은 정확한 임계값이 아니라 대략적인 벤치마크 역할을 하는 **목표 지향적 기준**입니다. 엄격함을 추구하되, 감각 기반 평가(vibes-based assessment)가 일부 포함될 수 있습니다. 더 체계적인 측정 가이드와 도구는 현재 개발 중입니다.

### 정량적 지표

- **관련 쿼리의 90%에서 스킬 트리거**: 스킬이 트리거될 만한 테스트 쿼리 10-20개를 실행한 뒤, 자동 로드 횟수 vs 수동 호출 횟수를 추적합니다.
- **X번의 도구 호출로 워크플로우 완료**: 스킬 유무에 따라 동일한 작업을 비교하고, 도구 호출 횟수와 총 토큰 소비량을 측정합니다.
- **워크플로우당 API 호출 실패 0건**: 테스트 실행 중 MCP 서버 로그를 모니터링하고, 재시도율과 에러 코드를 중점적으로 기록합니다.

### 정성적 지표

- **사용자가 다음 단계에 대해 Claude에게 프롬프트할 필요가 없음**: 테스트 중 방향 전환이나 명확화가 얼마나 자주 필요한지 기록합니다. 베타 사용자에게도 이 측면에 대한 피드백을 요청해야 합니다.
- **사용자 수정 없이 워크플로우가 완료됨**: 동일한 요청을 3-5번 실행하고, 구조적 일관성과 품질에 대해 출력을 비교합니다.
- **세션 간 일관된 결과**: 핵심 질문은 "새로운 사용자가 최소한의 안내로 첫 시도에 작업을 완수할 수 있는가?"입니다.

## 기술 요구사항

### 파일 구조

```
your-skill-name/
├── SKILL.md              # 필수 - 메인 스킬 파일
├── scripts/              # 선택 - 실행 가능한 코드
│   ├── process_data.py
│   └── validate.sh
├── references/           # 선택 - 문서
│   ├── api-guide.md
│   └── examples/
└── assets/               # 선택 - 템플릿 및 기타 에셋
    └── report-template.md
```

### 핵심 규칙

**SKILL.md 네이밍:**

- 파일 이름은 정확히 `SKILL.md`여야 합니다 (대소문자 구분)
- `SKILL.MD`나 `skill.md` 같은 변형은 허용되지 않습니다

**스킬 폴더 네이밍:**

- 폴더 이름은 kebab-case를 사용해야 합니다. 예: `notion-project-setup`
- 공백 허용 안 됨 (예: ~~`Notion Project Setup`~~)
- 밑줄 허용 안 됨 (예: ~~`notion_project_setup`~~)
- 대문자 허용 안 됨 (예: ~~`NotionProjectSetup`~~)

**README.md 금지:**

- 스킬 폴더 안에 `README.md` 파일을 포함하지 마세요
- 모든 문서는 `SKILL.md` 또는 `references/` 디렉토리에 배치해야 합니다
- GitHub을 통해 스킬을 배포할 때는, 사람 사용자를 위해 저장소 수준의 `README`를 두는 것이 권장됩니다 (자세한 내용은 "배포 및 공유" 섹션 참조)

## YAML frontmatter: 가장 중요한 부분

YAML frontmatter는 Claude에게 스킬을 로드할지 여부를 알려주기 때문에 매우 중요합니다.

### 최소 필수 형식

```yaml
---
name: your-skill-name
description: What it does. Use when user asks to [specific phrases].
---
```

frontmatter를 시작하기 위해 필요한 것은 이것이 전부입니다.

### 필드 요구사항

**name (필수):**

- kebab-case만 사용
- 공백이나 대문자를 포함하면 안 됨
- 스킬 폴더 이름과 일치해야 함

**description (필수):**

- 반드시 다음 두 가지가 모두 포함되어야 합니다:
    - 스킬이 **무엇을** 하는지에 대한 명확한 설명
    - **언제** 사용하는지에 대한 트리거 조건
- 1024자 미만이어야 함
- XML 태그 (`<` 또는 `>`)가 포함되어서는 안 됨
- 사용자가 스킬을 호출하기 위해 말할 수 있는 구체적인 작업 포함
- 관련이 있다면, 스킬이 처리하는 파일 유형 명시

## 효과적인 스킬 작성하기

### description 필드

Anthropic의 엔지니어링 블로그에 따르면: "이 메타데이터는 Claude가 각 스킬을 언제 사용해야 하는지 알 수 있도록 충분한 정보를 제공하면서도, 전체를 컨텍스트에 로드하지 않아도 되게 합니다." 이것이 점진적 공개(progressive disclosure)의 첫 번째 수준입니다.

**구조**: `[무엇을 하는지] + [언제 사용하는지] + [핵심 기능]`

**좋은 description 예시:**

```yaml
# 좋음 - 구체적이고 실행 가능함
description: Figma 디자인 파일을 분석하고 개발자 핸드오프 문서를 생성한다.
  사용자가 .fig 파일을 업로드하거나, "디자인 스펙", "컴포넌트 문서",
  "디자인-투-코드 핸드오프"를 요청할 때 사용한다.

# 좋음 - 트리거 문구 포함
description: 스프린트 계획, 태스크 생성, 상태 추적을 포함한 Linear 프로젝트
  워크플로우를 관리한다. 사용자가 "스프린트", "Linear 태스크", "프로젝트 계획"을
  언급하거나, "티켓 만들어줘"라고 요청할 때 사용한다.

# 좋음 - 명확한 가치 제안
description: PayFlow를 위한 엔드투엔드 고객 온보딩 워크플로우.
  계정 생성, 결제 설정, 구독 관리를 처리한다. 사용자가 "새 고객 온보딩",
  "구독 설정", "PayFlow 계정 만들어줘"라고 말할 때 사용한다.
```

**나쁜 description 예시:**

```yaml
# 너무 모호함
description: 프로젝트를 돕는다.

# 트리거 누락
description: 정교한 다중 페이지 문서 시스템을 생성한다.

# 너무 기술적이고 사용자 트리거 없음
description: 계층적 관계를 가진 Project 엔티티 모델을 구현한다.
```

### 선택적 필드

**License (선택):** 스킬을 오픈 소스로 공개할 경우 명시 (예: MIT, Apache-2.0)

**Compatibility (선택):** 글자 수 제한, 의도된 제품, 필요한 시스템 패키지, 네트워크 접근 필요 여부 같은 환경 요구사항

**Metadata (선택):** 임의의 키-값 쌍

```yaml
metadata:
  author: ProjectHub
  version: 1.0.0
  mcp-server: projecthub
```

### 보안 제한사항

**frontmatter에서 금지되는 것:**

- XML 꺾쇠 괄호 (`< >`)
- 이름에 "claude" 또는 "anthropic"이 포함된 스킬은 예약됨

**이유:** frontmatter는 Claude의 시스템 프롬프트에 나타나므로, 악의적인 콘텐츠가 잠재적으로 지시를 주입할 수 있습니다.

## 메인 지침 작성하기

frontmatter 이후에, 실제 지침은 Markdown으로 작성해야 합니다.

### 권장 구조

SKILL.md의 권장 구조는 다음과 같습니다:

> **frontmatter 영역** (`---`로 감싸기)
> - `name: your-skill`
> - `description: [...]`
>
> **본문 영역**
> - `# Your Skill Name` (제목)
> - `## Instructions` (지침)
> - `### Step 1: [첫 번째 주요 단계]` — 무엇이 일어나는지에 대한 명확한 설명
    >   - Example: `python scripts/fetch_data.py --project-id PROJECT_ID`
>   - Expected output: [성공이 어떤 모습인지 설명]
>   - (필요에 따라 더 많은 단계를 추가)
> - `## Examples`
    >   - `### Example 1: [일반적인 시나리오]`
>   - 사용자가 말합니다: "새 마케팅 캠페인을 설정해줘"
>   - Actions: 1) MCP를 통해 기존 캠페인을 가져온다 2) 제공된 매개변수로 새 캠페인을 생성한다
>   - Result: 확인 링크가 포함된 캠페인이 생성됨
> - `## Troubleshooting`
    >   - Error: [일반적인 에러 메시지]
>   - Cause: [발생 원인]
>   - Solution: [해결 방법]

---

## 지침 작성의 모범 사례

### 1. 구체적이고 실행 가능하게 작성하라

**좋은 예:**

> 데이터 형식을 확인하려면 `python scripts/validate.py --input {filename}` 명령을 실행하세요. 검증 실패 시 일반적인 문제: 필수 필드 누락(CSV 파일에 추가해야 함), 잘못된 날짜 형식(예상 형식은 YYYY-MM-DD)

**나쁜 예:**

> 진행하기 전에 데이터를 검증하라. (구체성과 실행 가능한 단계 부족)

### 2. 에러 처리를 포함하라

**MCP 연결 실패 시:**

"Connection refused" 에러가 발생하면:
1. MCP 서버가 실행 중인지 확인 — Settings > Extensions로 이동
2. API 키가 유효한지 확인
3. Settings > Extensions > [해당 서비스] > Reconnect로 재연결 시도

### 3. 번들된 리소스를 명확하게 참조하라

쿼리를 작성하기 전에, `references/api-patterns.md` 문서를 참조하세요:
- Rate limiting 가이드
- 페이지네이션 패턴
- 에러 코드 및 처리

### 4. 점진적 공개를 활용하라

- 메인 SKILL.md를 핵심 지침에 집중하도록 유지
- 더 상세한 문서는 `references/` 디렉토리로 옮기고 메인 지침에서 링크
- 이 3단계 시스템이 어떻게 작동하는지에 대한 더 깊은 이해를 위해서는 핵심 설계 원칙을 참조

# Chapter 3: 테스트와 반복

스킬 테스트는 상황에 맞게 강도를 조절할 수 있습니다. 크게 세 가지 방법이 있습니다:

- **Claude.ai에서 수동 테스트**: 직접 쿼리를 날려보며 스킬 동작을 확인합니다. 별도 설정 없이 빠르게 반복할 수 있습니다.
- **Claude Code에서 스크립트 테스트**: 테스트 케이스를 자동화해서 스킬 변경 시마다 일관된 검증을 수행합니다.
- **Skills API를 통한 프로그래밍 테스트**: 정의된 테스트 셋을 체계적으로 실행하는 평가 스위트를 구축합니다.

방법을 고를 때는 스킬의 품질 요구 수준과 사용 범위를 고려하세요. 소규모 팀이 내부에서 쓰는 스킬과 수천 명의 기업 사용자에게 배포하는 스킬은 테스트 기준이 다를 수밖에 없습니다.

> **프로 팁**: 테스트 범위를 넓히기 전에, 하나의 작업부터 완성하세요. 실력 있는 스킬 제작자들은 Claude가 까다로운 작업 하나를 성공적으로 수행할 때까지 반복한 뒤, 그 성공 패턴을 스킬로 추출합니다.

## 권장 테스트 접근 방식

### 1. 트리거 테스트

**목표:** 스킬이 적절한 시점에 로드되는지 확인

**트리거되어야 하는 경우:**

- "새 ProjectHub 워크스페이스 설정을 도와줘"
- "ProjectHub에서 프로젝트를 만들어야 해"
- "Q4 기획을 위한 ProjectHub 프로젝트를 초기화해줘"

**트리거되지 않아야 하는 경우:**

- "샌프란시스코 날씨가 어때?"
- "Python 코드 작성을 도와줘"
- "스프레드시트 만들어줘" (ProjectHub 스킬이 시트를 처리하지 않는 한)

### 2. 기능 테스트

**목표:** 스킬이 올바른 결과를 내는지 검증

**예시:**

- **테스트**: 작업 5개가 포함된 프로젝트 생성
- **주어진 조건**: 프로젝트 이름 "Q4 Planning", 작업 설명 5개
- **실행 시**: 스킬이 워크플로우 실행
- **그러면**:
    - ProjectHub에 프로젝트 생성됨
    - 올바른 속성의 작업 5개가 생성됨
    - 모든 작업이 프로젝트에 연결됨
    - API 오류 없음

### 3. 성능 비교

**목표:** 스킬 도입 전후로 결과가 얼마나 개선되는지 확인

| 지표 | 스킬 없이 | 스킬 사용 시 |
|---|---|---|
| 실행 방식 | 사용자가 매번 지시 | 자동 워크플로우 실행 |
| 왕복 메시지 | 15회 | 확인 질문 2회만 |
| API 호출 실패 | 3회 (재시도 필요) | 0회 |
| 토큰 소비 | 12,000 | 6,000 |

## skill-creator 스킬 사용하기

`skill-creator` 스킬은 Claude.ai 플러그인 디렉토리에서 바로 쓰거나, Claude Code용으로 다운로드할 수 있습니다. 스킬을 만들고 개선하는 과정을 도와줍니다.

- **스킬 생성**: 자연어 설명으로 스킬 생성, 올바른 형식의 SKILL.md 생성, 트리거 문구와 구조 제안
- **스킬 검토**: 흔한 문제점 감지, 트리거 과다/부족 위험 식별, 테스트 케이스 제안
- **반복 개선**: 엣지 케이스나 실패를 발견하면, 해당 사례를 skill-creator에 다시 넘겨 개선하세요

> **사용법**: "skill-creator 스킬을 사용해서 [내 사용 사례]를 위한 스킬을 만들어줘"

스킬은 살아있는 문서입니다. 피드백을 반영해 꾸준히 개선할 계획을 세우세요.

### 트리거 문제 진단

**트리거가 부족한 경우:**

- 스킬이 로드되어야 할 때 로드되지 않음
- 사용자가 수동으로 활성화해야 함
- **해결**: description에 세부 설명과 키워드를 추가

**트리거가 과도한 경우:**

- 관련 없는 쿼리에도 스킬이 로드됨
- 사용자가 스킬을 비활성화함
- **해결**: 부정 트리거를 추가하고 조건을 더 구체적으로 좁히기

**실행 문제:**

- 결과가 들쭉날쭉하거나, API 호출이 실패하거나, 사용자 수정이 필요함
- **해결**: 지침을 다듬고 에러 처리를 추가

# Chapter 4: 배포와 공유

스킬은 MCP 통합의 완성도를 높여줍니다. 사용자가 커넥터를 비교할 때, 스킬이 포함된 커넥터는 더 빠르게 가치를 전달하므로 MCP만 제공하는 대안보다 경쟁력이 높습니다.

## 현재 배포 모델 (2026년 1월)

**개별 사용자가 스킬을 설치하는 방법:**

1. 스킬 폴더를 다운로드
2. 필요하면 폴더를 압축
3. Claude.ai에서 설정 > 기능 > Skills를 통해 업로드
4. 또는 Claude Code의 Skills 디렉토리에 배치

**조직 수준 Skills:**

- 관리자가 워크스페이스 전체에 스킬을 배포 가능 (2025년 12월 18일 출시)
- 자동 업데이트
- 중앙 집중식 관리

## 오픈 스탠다드

Anthropic은 Agent Skills를 오픈 스탠다드로 공개했습니다. MCP와 마찬가지로, 스킬은 도구와 플랫폼에 구애받지 않아야 합니다 — 같은 스킬이 Claude든 다른 AI 플랫폼이든 동일하게 작동해야 합니다. 다만 특정 플랫폼의 기능을 최대한 활용하도록 설계된 스킬도 있으며, 이 경우 작성자가 호환성 필드에 명시할 수 있습니다.

## API를 통한 Skills 사용

애플리케이션, 에이전트, 자동화 워크플로우 등 프로그래밍 방식의 사용 사례에서는 API를 통해 스킬 관리와 실행을 직접 제어할 수 있습니다.

**주요 기능:**

- 스킬 목록 조회 및 관리를 위한 `/v1/skills` 엔드포인트
- `container.skills` 파라미터로 Messages API 요청에 스킬 추가
- Claude Console을 통한 버전 관리
- 커스텀 에이전트 구축을 위한 Claude Agent SDK 연동

**API와 Claude.ai, 어디서 Skills를 쓸까?**

| 사용 사례 | 최적의 환경 |
|---|---|
| 최종 사용자가 스킬과 직접 상호작용 | Claude.ai / Claude Code |
| 개발 중 수동 테스트 및 반복 | Claude.ai / Claude Code |
| 일회성, 임시 워크플로우 | Claude.ai / Claude Code |
| 앱에서 프로그래밍 방식으로 스킬 호출 | API |
| 대규모 프로덕션 배포 | API |
| 자동화 파이프라인 및 에이전트 시스템 | API |

> **참고**: API에서 Skills를 사용하려면 Code Execution Tool 베타가 필요합니다. 이 기능이 스킬 실행에 필요한 안전한 환경을 제공합니다.

## 현재 권장 접근 방식

1. **GitHub에 호스팅**
    - 오픈소스 스킬용 공개 리포지토리
    - 설치 방법이 담긴 명확한 README
    - 사용 예시 및 스크린샷

2. **MCP 리포지토리에 문서화**
    - MCP 문서에서 Skills로 링크 연결
    - 함께 사용할 때의 이점 설명
    - 빠른 시작 가이드 제공

3. **설치 가이드 작성**
    1. 스킬 다운로드 (리포지토리 클론 또는 Releases에서 ZIP 다운로드)
    2. Claude에 설치 (Claude.ai > 설정 > Skills에서 업로드)
    3. 스킬 활성화 (토글 켜기, MCP 서버 연결 확인)
    4. 테스트 (Claude에게 물어보기)

## 스킬 포지셔닝

**기능이 아닌 결과를 강조하세요:**

- **좋은 예**: "ProjectHub 스킬을 쓰면 수동 설정에 30분 걸리던 프로젝트 워크스페이스를 몇 초 만에 세팅할 수 있습니다."
- **나쁜 예**: "ProjectHub 스킬은 YAML 프론트매터와 Markdown 지시사항이 들어 있는 폴더로, MCP 서버 도구를 호출합니다."

**MCP + Skills 시너지를 부각하세요:**

"MCP 서버는 Claude에게 Linear 프로젝트 접근 권한을 부여합니다. Skills는 Claude에게 팀의 스프린트 계획 워크플로우를 가르칩니다. 이 둘이 합쳐져 AI 기반 프로젝트 관리가 완성됩니다."

# Chapter 5: 패턴과 트러블슈팅

아래 패턴들은 얼리 어답터와 내부 팀이 만든 스킬에서 뽑아낸 것입니다. 실제로 효과가 검증된 접근 방식이지, 반드시 따라야 할 틀은 아닙니다.

![skill-guide-infographic-06-patterns](https://raw.githubusercontent.com/joonlab/md-share-db/main/images/skill-guide-infographic-06-patterns_a81b5f.png)

## 접근 방식 선택하기: 문제 중심 vs. 도구 중심

홈디포(Home Depot)에 비유해 보겠습니다:

- **문제 중심**: "프로젝트 워크스페이스를 설정해야 해요" → 스킬이 적절한 MCP 호출을 올바른 순서로 조율합니다. 사용자는 원하는 결과만 말하고, 도구 선택은 스킬이 알아서 합니다.
- **도구 중심**: "Notion MCP가 연결되어 있어요" → 스킬이 Claude에게 최적의 워크플로우와 모범 사례를 알려줍니다. 사용자에게 접근 권한이 있고, 스킬이 전문 지식을 보태줍니다.

## 패턴 1: 순차적 워크플로우 오케스트레이션

**언제 쓰나:** 정해진 순서대로 진행해야 하는 다단계 프로세스가 필요할 때

**예시: 신규 고객 온보딩 워크플로우**

1. **계정 생성**: MCP 도구 `create_customer` 호출 (이름, 이메일, 회사 파라미터)
2. **결제 설정**: MCP 도구 `setup_payment_method` 호출 후 결제 수단 검증 대기
3. **구독 생성**: MCP 도구 `create_subscription` 사용 (`plan_id`와 1단계의 `customer_id`)
4. **환영 이메일 발송**: MCP 도구 `send_email` 호출 (`welcome_email_template` 사용)

**핵심 기법:**

- 단계 순서를 명시적으로 지정
- 단계 간 의존성 설정
- 각 단계마다 유효성 검증
- 실패 시 롤백 지침 포함

---

## 패턴 2: 멀티 MCP 협업

**언제 쓰나:** 워크플로우가 여러 서비스에 걸쳐 있을 때

**예시: 디자인에서 개발로 핸드오프**

1. **디자인 내보내기 (Figma MCP)**: 디자인 에셋 내보내기, 디자인 명세서 생성, 에셋 매니페스트 생성
2. **에셋 저장 (Drive MCP)**: 프로젝트 폴더 생성, 모든 에셋 업로드, 공유 링크 생성
3. **태스크 생성 (Linear MCP)**: 개발 태스크 생성, 에셋 링크 첨부, 엔지니어링 팀에 할당
4. **알림 (Slack MCP)**: #engineering 채널에 핸드오프 요약 게시, 에셋 링크와 태스크 참조 포함

**핵심 기법:**

- 단계별 명확한 분리
- MCP 간 데이터 전달
- 다음 단계로 넘어가기 전 유효성 검증
- 에러를 한곳에서 처리

## 패턴 3: 반복적 개선

**언제 쓰나:** 반복할수록 출력 품질이 좋아질 때

**예시: 보고서 생성**

1. **초기 초안**: MCP로 데이터 가져오기 → 첫 번째 초안 작성 → 임시 파일에 저장
2. **품질 검사**: 유효성 검사 스크립트 실행 (`scripts/check_report.py`) → 누락된 섹션, 일관성 없는 서식, 데이터 검증 오류 찾기
3. **개선 루프**: 각 문제 수정 → 해당 섹션 재생성 → 재검증 → 품질 기준 충족까지 반복
4. **마무리**: 최종 서식 적용 → 요약 생성 → 최종 버전 저장

**핵심 기법:**

- 명시적 품질 기준 설정
- 반복을 통한 점진적 개선
- 유효성 검사 스크립트 활용
- 반복 중단 시점 판단

- 대체 옵션 마련

## 패턴 5: 도메인 특화 인텔리전스

**언제 쓰나:** 단순한 도구 접근을 넘어 전문 지식까지 스킬에 담고 싶을 때

**예시: 컴플라이언스가 적용된 결제 처리**

1. **처리 전 (컴플라이언스 검사)**:
    - MCP로 거래 상세 정보 가져오기
    - 컴플라이언스 규칙 적용 (제재 목록 확인, 관할권 허용 여부, 위험 수준 평가)
    - 컴플라이언스 판단 내역 문서화

2. **처리**:
    - 컴플라이언스 통과 시: 결제 처리 MCP 호출, 사기 방지 검사 적용
    - 컴플라이언스 미통과 시: 거래를 검토 대상으로 표시, 컴플라이언스 케이스 생성

3. **감사 추적**:
    - 모든 컴플라이언스 검사 로깅
    - 처리 판단 내역 기록
    - 감사 보고서 생성

**핵심 기법:**

- 도메인 전문성을 로직에 녹여 넣기
- 실행 전 컴플라이언스 확인
- 빠짐없는 문서화
- 명확한 거버넌스 체계

## 트러블슈팅

### 스킬이 업로드되지 않을 때

**오류: "Could not find SKILL.md in uploaded folder"**

- 원인: 파일 이름이 정확히 SKILL.md가 아님
- 해결: SKILL.md로 이름을 바꾸세요 (대소문자 구분). `ls -la`로 확인

**오류: "Invalid frontmatter"**

- 원인: YAML 서식 문제
- 흔한 실수: `---` 구분자 누락, 닫히지 않은 따옴표
- 올바른 형식: `---`로 감싸고 그 안에 `name`과 `description` 작성

**오류: "Invalid skill name"**

- 원인: 이름에 공백이나 대문자가 들어감
- 잘못된 예: `My Cool Skill` → 올바른 예: `my-cool-skill`

### 스킬이 트리거되지 않을 때

**해결:** description 필드를 다듬으세요.

**빠른 체크리스트:**
- 너무 일반적이지 않은가?
- 사용자가 실제로 쓸 법한 트리거 문구가 들어 있는가?
- 관련 파일 유형이 언급되어 있는가?

**디버깅:** Claude에게 "[스킬 이름] 스킬은 언제 쓰나요?"라고 물어보세요. Claude가 description을 인용해서 답하므로, 빠진 내용을 파악해 보완하세요.

### 스킬이 너무 자주 트리거될 때

1. **제외 조건 추가**: "단순한 데이터 탐색에는 사용하지 마세요"
2. **더 구체적으로 쓰기**: "문서를 처리합니다" → "계약 검토를 위한 PDF 법률 문서를 처리합니다"
3. **범위 한정하기**: "전자상거래용 PayFlow 결제 처리. 일반 금융 질문에는 사용하지 않습니다."

### MCP 연결 문제

스킬은 정상 로드되는데 MCP 호출이 실패하는 경우:

1. **MCP 서버 연결 확인**: Settings > Extensions > 해당 서비스에서 "Connected" 상태인지 확인
2. **인증 확인**: API 키가 유효한지, 권한/스코프가 맞는지, OAuth 토큰이 만료되지 않았는지 점검
3. **MCP 단독 테스트**: 스킬 없이 Claude에게 직접 MCP 호출을 요청해 보세요. 여기서도 실패하면 스킬이 아니라 MCP 자체 문제입니다
4. **도구 이름 확인**: 스킬에서 참조하는 MCP 도구 이름이 정확한지 확인하세요 (대소문자 구분)

### 지침을 제대로 따르지 않을 때

**주요 원인과 해결법:**

1. **지침이 너무 장황함**: 간결하게 유지하고, 불릿 포인트를 쓰고, 상세 참조 자료는 별도 파일로 분리하세요
2. **지침이 묻혀 있음**: 중요한 지침을 상단에 배치하고, "## Important" 같은 헤더를 활용하세요
3. **모호한 표현**: "적절하게 검증하세요" → "CRITICAL: create_project를 호출하기 전에 프로젝트 이름이 비어 있지 않을 것, 최소 한 명의 팀 멤버가 할당되어 있을 것, 시작 날짜가 과거가 아닐 것을 확인하세요"
4. **모델의 "게으름"**: "시간을 들여 철저하게 수행하세요", "속도보다 품질이 중요합니다" 같은 명시적 지시를 추가하세요

> **고급 기법**: 중요한 유효성 검증이라면 언어 지침에만 의존하지 말고, 프로그래밍 방식으로 검사하는 스크립트를 함께 번들링하세요. 코드는 결정론적이지만, 자연어 해석은 그렇지 않습니다.

### 대용량 컨텍스트 문제

스킬이 느려지거나 응답 품질이 떨어지는 경우:

1. **SKILL.md 크기 최적화**: 상세 문서는 `references/` 폴더로 옮기고, 인라인 대신 참조 링크를 사용하세요. SKILL.md는 5,000단어 이하로 유지하세요
2. **활성화된 스킬 수 줄이기**: 20~50개 이상 동시에 활성화했는지 점검하세요. 선택적 활성화를 권장하며, 관련 기능끼리 묶은 스킬 "팩"을 고려해 보세요
